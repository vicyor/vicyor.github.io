apollo

发表于 2020-07-11 更新于 2020-07-12 阅读次数： Valine：

简介

灰度发布

1.application - 应用.
2.enviroment - 环境.
3.cluster - 集群.
4.namespace - 命名空间.

1.配置是独立于程序的只读变量.
常见的配置有：DB Connection Str、Thread Pool Size、Buffer Size、Request Timeout、Feature Switch、Server Urls等.
2.配置的加载方式.
程序内部hard code，配置文件，环境变量，启动参数，基于数据库.
3.配置的治理.
1.权限的控制. 对配置的修改必须有比较完善的权限控制.
2.不同环境、集群配置管理.
3.框架类组件配置管理. - cat客户端的配置.

1.统一管理不同环境、不同集群的配置.
namespace -> 支持多个不同的应用共享同一份配置，允许应用对共享的配置进行覆盖.
2.配置修改实时生效.
热发布
3.版本发布管理.
版本 -> 配置回滚.
4.灰度发布.
5.权限管理、发布审核、操作审计.
6.客户端配置信息监控.
7.java/.Net原生客户端.
Spring PlaceHolder,Annotation,SpringBoot的ConfigurationProperties.
8.提供开发平台API.
9.部署简单.

1.用户在配置中心对配置进行修改并发布.
2.配置中心通知Apollo客户端有配置更新.
3.Apollo客户端从配置中心拉取最新的配置,更新本地配置并通知到应用.

1.application.
标识 – appId.
2.environment.
环境 – Seystem Property 或 server.properties。
3.cluster.
集群 – 按照数据中心划分.
默认对应的配置为server.properties中的idc属性.
4.namespace.
一个应用下不同配置的分组.

1.同一个配置在不同时间点的值不同.
如商品如果卖的特别火爆,需要调整为每人限购2个.
2.传统应用配置问题.
1.配置使用静态配置,无法动态修改.
2.配置散乱,格式不标准.
xml/properties/DB.
3.容易发生生产事故.
将非生产的配置带到生产上,引发事故.
4.配置修改麻烦,周期长.
修改配置并动态部署后,特别费时费力.
5.配置信息缺少安全审计和版本控制功能.
3.配置中心对上述问题的解决.
1.集中式配置,所有配置信息存放在配置中心.
2.配置中心统一管理格式.
3.环境隔离,不同的环境对应不同的配置,互不干扰.
4.配置集中一次修改,实时通知所有客户端.
5.所有修改存在历史记录,方便查找谁和时间.
可按需回退到历史版本.
4.现在核心配置需求.
1.交互件和配置分离.
2.抽象标准化.
3.集中式.
4.高可用.
5.实时性.
6.治理.

1.静态配置.
1.环境相关.
数据库/中间件/其它服务的连接字符串.
2.安全配置.
用户名/密码/令牌/许可证书.
2.动态配置.
1.应用配置.
请求超时、线程池、队列、缓存、数据库连接池的容量、日志级别、限流熔断阈值、黑白名单.
2.功能开关.
蓝绿发布、灰度开关、降级开关，HA高可用开关，DB迁移.
蓝绿发布 -> 蓝色 -> Version N ;
绿色 -> Version N+1 ;
蓝色版本和绿色版本的完全切换(负载均衡、反向代理、路由).
灰度发布 -> 黑和白之间,平滑过渡(20%-80%-100%)的一种方式.
AB测试 -> 应用有2个或多个版本,让组成成分相同的访客群组(目标人群)随机访问这些版本.
收集各群组用户体验和业务数据,最后分析并评估出最好的版本并采用(或针对人群使用不同的A/B版本).
功能降级
DDOS攻击 -> (Distributed Denial of Service),处于不同位置的多个攻击者同时向一个或数个目标发动攻击.
或者一个攻击者控制了位于不同位置的多台机器并利用这些机器对受害者同时实施攻击.
DB/Schema迁移(Migration)
feature flag - 功能开关标志位
例子:
将数据从mongoDB迁移到Dynamo数据库.
状态 mongoDB Dynamo
初始状态 读写 100%
早期金丝雀写 读100% 。。。。。

。。。。。。

3.业务配置.
促销规则、贷款额度、利率等业务参数、A/B测试.
