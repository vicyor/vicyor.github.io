Apollo vs Spring Cloud Config

1.cluster 概念.
一个应用不同实例的分组,比如典型的多机房部署按照数据中心分集群.
2.Namespace是什么.
一个应用不同配置的分组,比如应用配置application,中间件配置framework,数据库配置database.
3.多个应用使用同一份配置,如何做到?
使用公有的Namespace.
4.客户端访问配置是否有权限,是否支持配置加密?
Apollo Client获取配置没有做权限控制,配置加密需要应用层自己实现.
md5 / 非对称加密(公私钥加密)
5.本地开发没有问题,为什么线上会出问题?
检查服务器上对应的data文件夹的读写权限,此权限对应的是程序运行时所对应的角色.
6.为啥不能进行编辑发布操作?
编辑和发布权限需要管理员授权,即使管理员自己也需要自己给自己授权.
7.接入了Apollo后值不更新怎么办?
1.通过 Config每次获取最新值.
2.监听配置变更事件.
3.Spring RefreshScope.
8.一个应用可以创建多个版本的配置吗?
使用集群Cluster.

1.@ConditionOnproperty -> 根据property动态配置.
2.@ConfigurationProperties -> 将配置映射到bean中.
3.PropertySource -> 配置源.
4.PropertySourcesPlaceholderConfigurer.
解析Bean的@Value的${…}参数占位符.

1.ApolloApplicationContextInitializer
加载application命名空间,并放入environment的propertysource中
2.PropertySourcesProcessor.
1.RemoteConfigRepository ->
两个定时任务:
1.longPollingRefresh. -> client http请求(long-polling)apollo的admin server获取配置更新的推送.
2.periodicRefresh. -> 客户端还会定时从Apollo配置中心服务端拉取应用的最新配置。
1.failback机制,防止long-polling失效导致配置更新失败.
2.客户端定时拉取apollo配置中心的配置(http request),若响应没有变化，一般服务器都会返回304(Not Modified).
2.LocalFileCofigRepository -> 将配置信息写到/opt/data目录.
3.将命名空间下的配置转换为PropertySource对象,放入spring的environment对象中. (environment主要用于初始化配置).
4.为propertySource的config添加AutoUpdateConfigChangeListener.
这个listener会在配置变化时候通过反射更新@Value注解对应的field(通过SpringValue找目标field).
3.PropertySourcesPlaceholderConfigurer(spring).
DefaultListableBeanfactory -> doCreateBean -> populateBean
会处理@Value注解,通过PropertySourcesPlaceholderConfigurer处理，将environment中的property注入.
4.SpringValueProcessor,SpringValueDefinitionProcessor.
会保留${key}和bean之间的关系.
5.ApolloAnnotationProcessor. -> 处理@ApolloConfigChangeListener注解，
处理流程->new ConfigChangeListener,该listener会通过反射调取注解所在的目标方法.并将listener注册到namespace对应config上.
